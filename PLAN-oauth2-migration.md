# OAuth2/OIDC Migration Plan for BV-BRC

## Context

BV-BRC currently uses a custom RSA-SHA1 pipe-delimited token format for authentication across all services. This plan migrates to standard OAuth2/OIDC, enabling third-party login (Google, ORCID, GitHub), browser-based CLI authentication (Device Authorization Grant), and industry-standard JWT tokens — while maintaining backward compatibility during the transition.

---

## Current Architecture Summary

### Token Format (custom, NOT JWT)
```
un=user@realm|tokenid=UUID|expiry=UNIX_TS|client_id=user@realm|token_type=Bearer|scope=user|[roles=...]|SigningSubject=URL|sig=RSA_SHA1_HEX
```
- Generated by `p3_user/generateToken.js` using RSA-SHA1 signing
- Validated by `p3_user/validateToken.js` — fetches public key from SigningSubject URL
- Duration: 24h (user), 744h (service)

### Identity Format
- MongoDB `users.id` = `jsmith` (bare username)
- Token `un` field = `jsmith@bvbrc` (username@realm)
- Solr `owner`/`user_read`/`user_write` fields = `jsmith@bvbrc`
- Workspace paths = `/jsmith@bvbrc/home`

### Services Affected
| Service | Role | Auth-relevant files |
|---------|------|-------------------|
| **p3_user** (port 3002) | Identity/auth, MongoDB user store | `generateToken.js`, `validateToken.js`, `middleware/token.js`, `routes/authenticate.js` |
| **p3_api** (port 3001) | Data API, Solr gateway | `middleware/auth.js`, `middleware/DecorateQuery.js`, `middleware/patch.js` |
| **bvbrc_website** (port 3000) | Web frontend | `public/js/p3/app/p3app.js`, `public/js/p3/widget/LoginForm.js` |
| **CLI** | Perl-based command-line tools | `POST /authenticate` with username/password |

## Technology Choice: `oidc-provider`

Use [`panva/node-oidc-provider`](https://github.com/panva/node-oidc-provider) — the only OpenID Certified Node.js OIDC implementation.

**Why:** Stays in the Node.js stack, supports all required grant types (Authorization Code + PKCE, Device Authorization, Client Credentials, Refresh Tokens), has a MongoDB adapter, and is actively maintained with conformance testing.

**Alternatives rejected:**
- Keycloak — Java, operational burden, different stack
- Auth0/Okta — vendor lock-in, cost, data sovereignty concerns for NIH-funded platform
- Custom JWT — not OIDC-certified, security risk of getting it wrong

---

## Service Architecture: New `p3_oidc` Service

Create a **new standalone service** rather than extending p3_user:

```
                    +-------------------+
                    |    p3_oidc        |  NEW — OIDC provider
                    | (oidc-provider)   |  Issues JWTs
                    | Port: 3003        |
                    +--------+----------+
                             |
                    Reads users, manages grants/sessions
                             |
                    +--------v----------+
                    |     MongoDB       |
                    | users (shared)    |
                    | + oidc_* colls    |
                    +-------------------+
```

**Rationale:**
- p3_user manages user records (CRUD, passwords, email verification) — distinct from authorization grant management
- Independent deployment and rollback
- p3_user continues serving legacy tokens during transition; p3_oidc issues JWTs
- `oidc-provider` expects its own Express app at a known issuer URL (e.g., `https://auth.bv-brc.org`)

## Critical Design Decision: JWT `sub` Claim

Set `sub` = `username@realm` (e.g., `jsmith@bvbrc`).

This is **non-negotiable** because `username@realm` is embedded throughout:
- Every Solr document's `owner`, `user_read`, `user_write` fields
- Every workspace path (`/username@realm/home`)
- Every permission API call

Using the same format means **zero changes** to p3_api's `DecorateQuery.js`, `patch.js`, `genomePermissionRouter.js`, or any Solr data.

### JWT Claims Structure
```json
{
  "iss": "https://auth.bv-brc.org",
  "sub": "jsmith@bvbrc",
  "aud": "bvbrc-api",
  "exp": 1700000000,
  "iat": 1699913600,
  "scope": "openid profile email",
  "preferred_username": "jsmith",
  "email": "jsmith@example.com",
  "roles": ["admin"],
  "realm": "bvbrc"
}
```

## OAuth2 Flows

### Web: Authorization Code + PKCE
1. User clicks Login → website redirects to `https://auth.bv-brc.org/auth?response_type=code&client_id=bvbrc-web&redirect_uri=...&code_challenge=...&scope=openid+profile+email`
2. p3_oidc presents login form (BV-BRC credentials or social login)
3. On success, redirects to `/callback?code=...`
4. Website backend exchanges code for JWT access token + refresh token
5. Social login buttons (Google, ORCID, GitHub) are choices on the p3_oidc login page, not on the website itself

### CLI: Device Authorization Grant (RFC 8628)
1. CLI calls `POST /device/authorize` → gets `device_code`, `user_code`, `verification_uri`
2. CLI displays: "Go to https://auth.bv-brc.org/device and enter code: ABCD-1234"
3. User authenticates in browser (any method including social login)
4. CLI polls `POST /token` → receives JWT + refresh token
5. Stores in `~/.bvbrc/auth_token`

**Advantage over current approach:** No passwords on disk, supports MFA, works with social login.

### Service-to-Service: Client Credentials
- Replaces current `POST /authenticate/service` (non-standard app-token + user-token exchange)
- Services present `client_id`/`client_secret` to get service-scoped JWT

### Job Submission: Token Exchange (RFC 8693)

Jobs may execute minutes, hours, or days after submission. Short-lived user tokens will be expired by then. The app service uses **Token Exchange** to mint fresh delegated tokens at execution time.

**At job submission:**
- The app service validates the user's access token, extracts the user identity (`jsmith@bvbrc`)
- Generates a **one-time job startup ticket** (cryptographically random string, e.g., `openssl rand -hex 32`)
- Stores the user identity and the ticket (hashed) in the scheduler DB with the job metadata
- The plaintext ticket is passed to Slurm as part of the job's environment or metadata

**At job execution (startup script):**

Token minting is deferred to job start time, not submission time. This avoids the unpredictable queue wait consuming token lifetime. The token request is added to the existing Slurm batch startup template (`app_service/lib/Bio/KBase/AppService/slurm_batch.tt`), which already handles job initialization:

1. Startup script reads the one-time ticket and job ID from job metadata
2. Script redeems the ticket at the app service's internal token endpoint:
   ```
   POST https://app-service.internal/api/job-token
   Body: { "ticket": "abc123...", "job_id": "uuid" }
   ```
3. The app service performs Token Exchange with p3_oidc using its client credentials:
   ```
   POST https://auth.bv-brc.org/token
   grant_type=urn:ietf:params:oauth:grant-type:token-exchange
   subject_token=jsmith@bvbrc
   subject_token_type=urn:ietf:params:oauth:token-type:access_token
   client_id=bvbrc-app-service
   client_secret=<secret>
   scope=user
   ```
4. p3_oidc issues a delegated JWT with lifetime matching the job's wall-time limit (plus buffer):
   ```json
   {
     "sub": "jsmith@bvbrc",
     "act": { "sub": "bvbrc-app-service" },
     "scope": "user",
     "exp": "<wall_time + 1 hour buffer>"
   }
   ```
5. Startup script sets the token as an environment variable (e.g., `BVBRC_AUTH_TOKEN`) for the job process
6. Job runs with a fresh token scoped to its actual execution window

**App service internal token endpoint:**
- `POST /api/job-token` — not public, reachable only from compute nodes on the internal network
- Authenticated via **one-time job startup ticket** — no shared secret or mTLS needed on compute nodes
- Validation: app service verifies the ticket hash matches the stored hash for the given job ID, and that the ticket has not already been redeemed
- On success: marks the ticket as consumed (single-use), performs Token Exchange with p3_oidc, returns the delegated JWT
- On failure (invalid ticket, already consumed, job not found): returns 403, logs the attempt
- A compromised compute node can only obtain tokens for jobs whose tickets it possesses — it cannot fabricate tickets for other jobs or other users
- Logs all token issuance and failed attempts (job ID, user, requested lifetime) for audit

**Token lifetime for jobs:**
- Lifetime is derived from the Slurm wall-time limit (`--time`) plus a buffer (e.g., 1 hour)
- p3_oidc enforces a per-client maximum lifetime for delegated tokens (e.g., 7 days for `bvbrc-app-service`)
- No refresh mechanism needed on compute nodes — the token lives long enough for the job to complete
- If a job exceeds the maximum token lifetime, it fails and must be resubmitted (edge case, matches current operational expectations)
- The client secret for Token Exchange stays on the app service — it is never distributed to compute nodes
- This is a clear improvement over the current 31-day service tokens: job tokens are scoped to actual execution time, not a fixed 744-hour window

**Advantages over storing a long-lived user token:**
- No bearer token stored at rest in the scheduler DB — only the username. If the DB is compromised, the attacker gets usernames, not tokens
- Tokens are minted fresh at execution time with appropriate lifetime
- User revocation works — if the account is disabled between submission and execution, p3_oidc refuses to issue a delegated token
- Audit trail — the `act` claim records that the app service made the call, not the user directly
- Job cancellation can trigger token revocation (p3_oidc revokes the specific delegated token by `tokenid`)

**Access control:** Only confidential clients on an explicit allowlist in p3_oidc can use Token Exchange. The `bvbrc-app-service` client is registered with `grant_types: ["client_credentials", "urn:ietf:params:oauth:grant-type:token-exchange"]`.

**Slurm security considerations:**

Between submission and execution, the one-time ticket sits in Slurm's job metadata (environment variables). Slurm stores this in its controller's internal state, its state save directory on disk (`StateSaveLocation`), and potentially its accounting database if job environment logging is enabled. This is a trust boundary worth noting:

- *Slurm admin access:* Users with Slurm admin privileges (`SlurmUser`, root on slurmctld host) can read environment variables of any job via `scontrol show job -d`. They could extract tickets for pending jobs. However, each ticket is single-use — redemption is logged and a second attempt is rejected.
- *User visibility:* By default, Slurm allows job owners to see their own environment but not other users'. Set `PrivateData=environment` in `slurm.conf` to enforce this restriction.
- *Compute node exposure:* The ticket is visible in `/proc/<pid>/environ` while the startup script runs. Any process running as the same user (or root) on the compute node could read it. But by the time the job is running, the ticket has already been redeemed and consumed — reading it after that point is useless.
- *Slurmctld host:* The controller host and its state directory should have restricted access (root only, same security posture as today).

*Comparison to current state:* Today, a full 31-day bearer token sits in Slurm job metadata. The one-time ticket is a significant improvement:
- Single-use: even if stolen from Slurm state, it can only be redeemed once
- If an attacker redeems it before the job starts, the job fails (visible to operators), and the attacker only gets a token scoped to the job's wall-time (hours, not 31 days)
- Theft is detectable: the legitimate job startup fails with a "ticket already consumed" error, which is logged and can trigger an alert

## Third-Party Provider Integration

Third-party providers are **upstream IdPs in p3_oidc**, not integrated directly into the website:

```
User → Website → p3_oidc login page → [BV-BRC | Google | ORCID | GitHub]
                                              ↓
                                   Account linking resolution
                                              ↓
                                   JWT with sub=user@bvbrc
```

### Account Types

**Passwordless accounts are first-class.** Users arriving via social login never need to set a BV-BRC password. The `password` field in the `users` collection remains optional (registration without password is already supported via the reset code flow). A user authenticates entirely through their social provider(s). They only need a BV-BRC password if they want to use the legacy username/password login. The CLI device flow works for passwordless users since it authenticates via browser.

A user can link **multiple social providers** to one BV-BRC account (e.g., Google + ORCID + GitHub). Each resolves to the same `sub = username@bvbrc`.

### Account Linking

**New MongoDB collection: `federated_identities`**
```json
{
  "bvbrc_user_id": "jsmith",
  "provider": "google",
  "provider_sub": "1234567890",
  "provider_email": "jsmith@gmail.com",
  "linked_at": ISODate,
  "last_used": ISODate
}
```
Indexes: `{ provider, provider_sub }` (unique), `{ bvbrc_user_id, provider }`

**Returning social login (linked account exists):** Lookup `federated_identities` → issue JWT directly. No interaction needed.

**First-time social login (no linked account):** p3_oidc authenticates with the social provider, checks `federated_identities` for that `{ provider, provider_sub }` — no match found. Then:

1. **No email collision** — provider's verified email does not match any existing BV-BRC account:
   - Present an account creation form pre-filled from the social provider's profile (name, email)
   - The user chooses their BV-BRC username (with availability check and same validation rules as current registration: alphanumeric, dot, dash, underscore)
   - Create the BV-BRC account (no password) with the chosen username
   - Create the `federated_identities` link
   - Issue JWT with `sub = chosen_username@bvbrc`

2. **Email collision** — provider's verified email matches an existing BV-BRC account:
   - Inform the user that an account with that email already exists
   - **If existing account has a password:** require the BV-BRC password to prove ownership before linking
   - **If existing account is passwordless** (social-only): send a verification code to the email on file, require the user to enter it to prove ownership
   - **Email match alone is never sufficient to auto-link** — this prevents an attacker from creating a social account with a victim's email and hijacking their BV-BRC account

**Linking from account settings (already logged in):** User goes to account settings, clicks "Link Google/ORCID/GitHub account," completes the social provider's OAuth flow, creates the `federated_identities` record.

**Account settings UI for linked accounts:** The user profile/settings page includes a "Linked Accounts" section showing:
- Which providers are currently linked (Google, ORCID, GitHub) with the associated email/ID
- An "Unlink" button for each linked provider
- "Link" buttons for providers not yet linked
- Guard: if the account is passwordless and only one provider is linked, prevent unlinking (would lock the user out). Prompt the user to either set a password or link another provider first.

**ORCID note:** Store ORCID ID (`0000-0002-1825-0097`) on the `users` document as `orcid_id` in addition to the federated identity link — valuable metadata for the research community.

## Dual-Token Validation in p3_api

During transition, `p3_api/middleware/auth.js` accepts both formats:

- **Legacy tokens** contain `|` characters
- **JWTs** contain exactly two `.` characters (header.payload.signature)

Both paths set `req.user = "username@realm"` — downstream middleware (`DecorateQuery.js`, `patch.js`, etc.) is unchanged.

JWT verification uses JWKS from `https://auth.bv-brc.org/.well-known/openid-configuration` with TTL-based caching (similar to existing `ssCache` for legacy public key).

## Phased Migration

### Phase 1: Deploy p3_oidc (Foundation)
- Create `p3_oidc` service with `oidc-provider`, MongoDB adapter, `findAccount` reading existing `users` collection
- Password validation reuses existing bcrypt/SHA1 dual logic from `p3_user/models/user.js`
- Deploy at `https://auth.bv-brc.org`
- Pre-register OAuth2 clients
- **Delivers:** Running OIDC provider, no user-facing changes, no risk to existing services

### Phase 2: Dual-token validation in p3_api
- Update `p3_api/middleware/auth.js` to detect and validate JWTs via JWKS
- Add `jose` library dependency
- **Delivers:** API accepts JWTs — any client can start using them while legacy tokens still work
- **Critical test:** `req.user` is identical for same user regardless of token format

### Phase 3: Website OIDC login
- Add `/callback` route to website backend for code exchange
- Update `LoginForm.js` — redirect-based OIDC flow with social login buttons
- Update `p3app.js` — JWT-aware `login()`, `checkLogin()`, token refresh via `grant_type=refresh_token`
- Update `Authorization` headers from raw token to `Bearer <jwt>`
- Add "Linked Accounts" section to `UserProfileForm.js` — view/link/unlink social providers
- **Delivers:** Users can log in via OIDC including Google/ORCID/GitHub, and manage their linked accounts

### Phase 4: CLI Device Authorization
- Enable Device Authorization Grant in p3_oidc
- Create `bvbrc-login` CLI command for device flow
- Update Perl `P3AuthToken` module to handle JWT format
- **Delivers:** CLI users authenticate via browser, no passwords in terminal

### Phase 5: Service-to-service migration
- Register service clients, migrate workspace/app services to Client Credentials
- Deprecate `POST /authenticate/service`
- **Delivers:** Standardized service auth

### Phase 6: Legacy deprecation (6+ months after Phase 3)
- Add deprecation warnings to legacy token validation
- Monitor usage, remove legacy support when safe
- **Delivers:** Single token format, simplified codebase

## Security Considerations

- **PKCE mandatory** for all Authorization Code flows (S256 only)
- **Refresh token rotation** — each refresh yields new token; old one invalidated; reuse triggers grant revocation
- **Token storage** — short-lived access tokens (15-30 min) in memory/localStorage, refresh tokens with sliding window (30 days) and absolute max (90 days)
- **RSA-SHA1 → RS256** — significant cryptographic upgrade (also addresses FIPS 140-2/SC-13 — SHA-1 is deprecated for digital signatures under NIST SP 800-131A)
- **Rate limiting** on `/token` and `/device/authorize` endpoints

## Key Management Procedures

### JWT Signing Keys (RSA Key Pairs)

**Key type:** RSA 2048-bit minimum (RS256). Consider RSA 4096 or EC P-256 (ES256) for new deployments.

**Generation:**
- Generated by a designated system administrator on the p3_oidc server (or a secure workstation)
- `openssl genrsa -out private.pem 2048 && openssl rsa -in private.pem -pubout -out public.pem`
- Generated keys are placed in p3_oidc's config directory with permissions `600` (owner-read-only), owned by the service user
- Key generation is logged (who, when, on what host) in the operations log

**Storage:**
- Private keys stored on the p3_oidc server filesystem only, never in source control, never in MongoDB
- Protected by file permissions (`chmod 600`) and OS-level full-disk encryption (LUKS or AWS EBS encryption)
- Backups follow the same encryption-at-rest requirements as the live keys

**Rotation schedule:** Quarterly (every 90 days), or immediately if compromise is suspected.

**Rotation procedure:**
1. Generate a new key pair on the p3_oidc server
2. Add the new key to p3_oidc's configuration as the **active signing key**
3. The old key remains in the JWKS endpoint as a **verification-only key** (published but not used for signing)
4. p3_api and other consumers cache JWKS with a TTL (1 hour) — they pick up the new key automatically
5. After all tokens signed with the old key have expired (access token lifetime + buffer, typically 24 hours), remove the old key from JWKS
6. Securely delete the old private key: `shred -u old-private.pem`
7. Log the rotation event (date, operator, old key fingerprint, new key fingerprint)

**Emergency rotation (key compromise):**
1. Generate new key pair immediately
2. Replace the signing key in p3_oidc, restart the service
3. Optionally remove the compromised key from JWKS immediately (this invalidates all outstanding tokens — users must re-authenticate)
4. Notify affected users if data exposure is possible
5. File an incident report per organizational policy

### OAuth2 Client Secrets

**Scope:** Confidential clients only (`bvbrc-app-service`, `bvbrc-workspace`). Public clients (`bvbrc-web`, `bvbrc-cli`) do not have secrets.

**Generation:**
- Generated using `openssl rand -base64 32` (256 bits of entropy)
- Stored bcrypt-hashed in MongoDB `oidc_clients` collection
- Plaintext stored in the service's config file on the server where the client runs, with `600` permissions

**Rotation schedule:** Annually, or immediately if compromise is suspected.

**Rotation procedure:**
1. Generate a new client secret
2. Update the `oidc_clients` record in MongoDB with the new bcrypt hash
3. Update the client service's config file with the new plaintext secret
4. Restart the client service
5. Verify the client can authenticate to p3_oidc
6. Log the rotation event

**Note:** `oidc-provider` supports multiple active secrets per client during rotation — add the new secret, update the client config, then remove the old secret.

### Third-Party Provider Credentials (Google, ORCID, GitHub)

**Scope:** OAuth2 client ID and client secret issued by each provider.

**Generation:** Created in each provider's developer console. These are not generated by BV-BRC.

**Storage:** In p3_oidc's config file on the server, `600` permissions. Never in source control.

**Rotation schedule:** Annually, or if compromised. Rotation requires generating new credentials in the provider's developer console and updating p3_oidc's config.

### Legacy RSA Signing Key (p3_user)

**During transition:** The existing RSA-SHA1 key pair in p3_user continues to sign legacy tokens. No rotation is needed for this key since it is being deprecated. It will be decommissioned in Phase 6.

**Decommission procedure:**
1. Confirm no legacy tokens are in active use (monitor logs for legacy token validation in p3_api)
2. Remove the private key from p3_user: `shred -u private.pem`
3. Remove the public key endpoint (`/public_key`)
4. Log the decommission event

### MongoDB Configuration Requirements

**Sensitive data in MongoDB:** The OIDC collections store refresh tokens (hashed), device codes, authorization codes, session data, and grant records. User passwords and client secrets are stored as bcrypt hashes (not recoverable). Refresh tokens are the primary concern — if stolen, an attacker can mint access tokens until revocation.

**Encryption at rest (SC-28):**
- Encrypt MongoDB data volume(s) with LUKS, key file stored on the boot disk (`/root/mongodb-luks.key`, permissions `400`) referenced in `/etc/crypttab` for auto-unlock at boot
- This protects against individual disk theft (decommissioned or failed drives) — the standard threat model for colo environments
- Boot-disk encryption is not required but recommended (eliminates auditor questions about config files and swap)
- Disable swap on database and application servers, or encrypt the swap partition — prevents MongoDB process memory (containing in-flight tokens) from being written to unencrypted disk
- Add drive wiping (`shred` or `nwipe`) to the hardware decommission checklist

**Backup impact:**
- Volume snapshots of a LUKS-encrypted volume capture encrypted blocks — existing snapshot-based MongoDB backup workflow continues to work unchanged
- Snapshots are encrypted at rest by default, which is a security benefit
- Restore adds one step: `cryptsetup luksOpen` with the key file before mounting, then start MongoDB as normal
- **Critical:** the LUKS key file (`/root/mongodb-luks.key`) must be backed up separately from the data volume — without it, snapshots are unrecoverable. Store a copy in a secure offline location (password manager, safe, or separate encrypted media)
- Additionally, add a recovery passphrase to the LUKS volume (`cryptsetup luksAddKey /dev/sdX`) as a secondary unlock method in case the key file is lost
- If using `mongodump` (logical backups) instead of snapshots: dump files are unencrypted (MongoDB reads decrypted data from the open volume) — store them on an encrypted volume or encrypt with `gpg` before offsite transfer

**Encryption in transit:**
- MongoDB connections must use TLS (`tls=true` in connection string)
- Verify current `mongo.url` config in p3_user and p3_oidc — if no `tls=true`, the connection is unencrypted

**Authentication and access control:**
- MongoDB must require authentication (not open access on localhost)
- p3_oidc should use a dedicated MongoDB user scoped to only the databases/collections it needs
- Separate MongoDB users for p3_user and p3_oidc where possible

**Network access:**
- MongoDB must not be exposed to the internet — bind to localhost or private network only
- Firewall to allow connections only from application servers

**TTL cleanup:**
- `oidc-provider`'s MongoDB adapter creates TTL indexes on its collections — expired tokens, codes, and sessions are automatically deleted
- This limits exposure from DB backups: leaked backups only contain tokens valid at backup time, which expire naturally

### Secrets Not in Source Control

The following files contain secrets and must remain in `.gitignore` (or equivalent):
- `p3-oidc.conf` (JWT signing key path, client secrets, provider secrets)
- `p3-user.conf` (legacy signing key path, MongoDB credentials)
- `p3api.conf` (Solr credentials)
- `*.pem` (all key material)

### Audit Requirements (NIST SP 800-53 AU-2/AU-3)

Log the following events:
- Key generation, rotation, and destruction (operator, timestamp, key fingerprint)
- Client secret creation and rotation
- Failed authentication attempts to p3_oidc
- Token Exchange requests (which service requested delegation for which user)
- Administrative actions (client registration changes, user role modifications)

## Future: User Groups and Group-Based Permissions

The OAuth2 migration does not change the authorization model — per-document `owner`, `user_read`, `user_write` fields in Solr remain as-is. However, the new JWT infrastructure is designed to support group-based permissions as a follow-on initiative.

### Current Limitation
Permissions are per-user: sharing a genome with 10 collaborators requires adding 10 individual usernames to `user_read`/`user_write`. There is no concept of a group.

### What Groups Would Enable
- Share data with a named group (e.g., `niaid-flu-team`) instead of individual users
- Manage group membership separately from per-resource permissions
- Adding/removing a team member automatically updates their access to all shared resources

### How OAuth2 Enables This
The JWT `groups` claim (reserved as an empty array in the initial rollout) can carry group memberships:
```json
{
  "sub": "jsmith@bvbrc",
  "groups": ["niaid-flu-team", "ncbi-collab"]
}
```

`DecorateQuery.js` would expand its filter to include group-based access:
```
&fq=(public:true OR owner:USER OR user_read:USER OR group_read:GROUP1 OR group_read:GROUP2)
```

Groups are resolved at login time and cached in the token. With short-lived access tokens (15-30 min), membership changes propagate quickly without requiring token revocation.

### What the Groups Initiative Requires (not in scope for OAuth2 migration)
- **MongoDB:** new `groups` collection (group_id, display_name, owner, members)
- **Solr schema:** add `group_read` and `group_write` multi-valued fields to all private-data collections (genome, genome_feature, genome_sequence, pathway, sp_gene, subsystem, genome_amr, genome_typing)
- **Reindexing:** existing Solr documents need the new fields (can default to empty)
- **p3_api:** update `DecorateQuery.js` to include group-based filters from JWT claims; update `genomePermissionRouter.js` to support granting access to groups
- **p3_oidc:** `findAccount` adapter extended to look up group memberships and include them in JWT claims
- **Website:** group management UI (create group, add/remove members) and sharing dialog updated to support groups alongside individual users

### Design Decisions to Make for Groups (deferred)
- Can a group own data, or only users?
- Can groups be nested (group of groups)?
